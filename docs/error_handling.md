# Symbiauth Error Handling Strategy

## Overview

Symbiauth uses structured error handling with **clear error classes**, **retry budgets**, **idempotency**, and **explicit timeout**s to ensure predictable failure modes.

---

## Error Classes

All errors map to **HTTP-style codes** for consistency across components:

| Code | Name | Scenario | Retry? | Example |
|------|------|----------|--------|---------|
| **400** | client_misuse | Invalid request format or parameter | No | Malformed JSON, missing `origin` |
| **401** | auth_required | Face ID approval needed (step-up recoverable) | Wait for user | Auth token missing/expired, step-up needed |
| **403** | policy_denied | Policy explicitly denies action (non-recoverable without config change) | No | Origin blocked, never allowed by policy |
| **404** | route_not_found | Unknown API route or resource type | No | Unknown message type, invalid endpoint |
| **409** | conflict | State conflict (e.g., already paired) | No | Duplicate pairing attempt |
| **408** | timeout | Operation exceeded deadline | Retry with backoff | iOS didn't respond in 30s |
| **429** | too_many_requests | Rate limit exceeded | Retry after delay | Token bucket exhausted |
| **500** | internal | Unexpected agent error | Maybe retry | Panic, unwrap failure |
| **503** | unavailable | Agent overloaded or db locked | Retry with backoff | Bounded queue full |

**Error Response Format**:
```json
{
  "type": "error",
  "err_code": 401,
  "err_reason": "auth_required",  // Stable machine-readable string
  "message": "Face ID approval needed",  // Human-readable
  "corr_id": "abc123"
}
```

**Notes**:
- **401 vs 403**: Use 401 for "can be fixed by user action now" (e.g., approve Face ID); 403 for "policy forbids this, period"
- **404 usage**: For vault/cred reads, return `{"status": "miss"}` with 200, not 404. Reserve 404 for unknown API routes.
- **err_reason**: Stable identifier (e.g., `auth_required`, `rate_limited`, `policy_denied`) for client logic; don't parse `message`

## Retry Policy

### **Idempotent Reads** - Retry with Jitter

```rust
async fn read_with_retry(key: &str, max_retries: u32) -> Result<Value> {
    for attempt in 0..max_retries {
        match vault.read(key).await {
            Ok(v) => return Ok(v),
            Err(e) if e.is_retryable() => {
                let backoff = Duration::from_millis(100 * 2_u64.pow(attempt));
                let jitter = rand::thread_rng().gen_range(0..backoff.as_millis() as u64);
                tokio::time::sleep(backoff + Duration::from_millis(jitter)).await;
            }
            Err(e) => return Err(e),
        }
    }
    Err(Error::Timeout)
}
```

**Retryable errors**: `503 unavailable`, `408 timeout`  
**Max retries**: 3  
**Backoff**: Exponential (100ms, 200ms, 400ms) + jitter  

### **Writes** - At-Most-Once with Idempotency Key

```rust
async fn write_idempotent(key: &str, value: &[u8], idempotency_key: &str) -> Result<()> {
    // Check if already applied (persisted in SQLite)
    if vault.was_applied(idempotency_key).await? {
        return Ok(()); // Already done, safe to return success
    }
    
    // Apply write
    vault.write(key, value).await?;
    vault.mark_applied(idempotency_key, Instant::now()).await?;
    Ok(())
}
```

**Idempotency keys**:
- UUID per request (generated by caller)
- **Persisted in SQLite** table `idempotency_keys(key TEXT PRIMARY KEY, created_at INTEGER)`
- TTL: Keys expire after 5 minutes (GC via scheduled task)
- **Survives agent crash**: Prevents duplicate writes after restart

**No retries on conflict**: Caller must decide whether to retry

## Timeout Guidelines

| Operation | Timeout | Rationale |
|-----------|---------|-----------|
| **Auth (Face ID)** | 30s | User needs time to lift phone, approve Face ID |
| **Vault read/write** | 2s | Local SQLite should be fast; longer suggests lock contention |
| **Native Messaging (first call)** | 3s | MV3 service worker cold-start; adaptive thereafter |
| **Native Messaging (warm)** | 1s | After first call, worker is warm; should respond quickly |
| **Proximity heartbeat** | 5s | TLS roundtrip + BLE check; longer suggests disconnection |
| **TLS handshake** | 10s | mTLS cert validation, network latency |

**Adaptive NM timeout**:
```rust
struct NMClient {
    first_call: bool,
}

impl NMClient {
    fn timeout(&self) -> Duration {
        if self.first_call {
            Duration::from_secs(3) // Cold-start
        } else {
            Duration::from_secs(1) // Warm
        }
    }
}
```

**Implementation**:
```rust
tokio::time::timeout(Duration::from_secs(30), auth_gate.request(corr_id))
    .await
    .map_err(|_| Error::Timeout)?
```

## Continuation Hygiene (iOS)

**Rule**: Every async operation on iOS **must** resolve its continuation (success, timeout, or cancel).

### **Example: Auth Request**

```swift
func handleAuthRequest(corrId: String) async throws {
    do {
        let proof = try await FaceIDAuthenticator.generateAuthProof(corrId: corrId, ...)
        try await tlsClient.send(proof)
        // ✅ Success path
    } catch FaceIDAuthenticator.AuthError.cancelled {
        // ✅ User cancelled - send cancel message to agent
        try? await tlsClient.send(["type": "auth.cancel", "corr_id": corrId])
    } catch {
        // ✅ Error path - agent times out, no orphaned Promise
        throw error
    }
}
```

**Anti-pattern**:
```swift
// ❌ BAD - continuation never resolves if backgrounded
Task {
    let proof = try await FaceIDAuthenticator.generateAuthProof(...) // Hangs if app backgrounds
}
```

**Correct pattern**:
```swift
// ✅ GOOD - always resolve, even on background
Task {
    guard UIApplication.shared.applicationState == .active else {
        sendCancelMessage()
        return
    }
    // ... generate proof
}
```

## Error Context & Logging

All errors include **context** for debugging:

```rust
#[derive(Debug, thiserror::Error)]
enum Error {
    #[error("vault read failed for key={key}: {source}")]
    VaultRead {
        key: String,
        #[source]
        source: VaultError,
    },
    
    #[error("auth timeout for corr_id={corr_id} after {elapsed:?}")]
    AuthTimeout {
        corr_id: String,
        elapsed: Duration,
    },
}
```

**Logging**:
```rust
tracing::error!(
    key = %key,
    err = %e,
    corr_id = %corr_id,
    "vault read failed"
);
```

**Log fields** (NDJSON):
- `ts`: ISO timestamp
- `level`: error, warn, info, debug
- `component`: vault, router, auth_gate, etc.
- `corr_id`: Request correlation ID
- `origin`: Website origin (if applicable)
- `action`: cred.get, vault.write, etc.
- `decision`: allow, deny, step_up
- `latency_ms`: Operation duration
- `err_code`: 400, 401, etc.

## Panic Handling

**Goal**: Never panic in production; use `Result` for recoverable errors.

### **Acceptable Panics**

1. **Startup configuration errors** (missing Keychain, corrupt vault)
   ```rust
   let db = open_vault().expect("vault must exist");
   ```
   **Rationale**: Can't operate without vault; fail-fast is correct.

2. **Invariant violations** (should never happen)
   ```rust
   assert!(tokens >= 0.0, "token bucket cannot go negative");
   ```
   **Rationale**: Logic bug; panic helps catch it in dev/test.

### **Unacceptable Panics**

1. **User input** (never `unwrap()` on user data)
   ```rust
   // ❌ BAD
   let origin = req.get("origin").unwrap();
   
   // ✅ GOOD
   let origin = req.get("origin").ok_or(Error::MissingField("origin"))?;
   ```

2. **External errors** (network, filesystem)
   ```rust
   // ❌ BAD
   tokio::fs::read(path).await.unwrap();
   
   // ✅ GOOD
   tokio::fs::read(path).await.map_err(Error::FileRead)?;
   ```

## Circuit Breaker (Future)

> [!NOTE]
> Not implemented in Phase 1-3; planned for Phase 4+.

**Purpose**: Temporarily disable flapping components (e.g., vault if SQLite is corrupt).

```rust
struct CircuitBreaker {
    state: State,
    failure_count: u32,
    threshold: u32,
    reset_timeout: Duration,
}

enum State {
    Closed,  // Normal operation
    Open,    // All requests fail-fast
    HalfOpen, // Test if recovered
}
```

**Example use**:
- If vault.read fails 5 times in a row → Open circuit
- Return `503 unavailable` without trying vault
- After 30s → HalfOpen, try one request
- If success → Close; if fail → Open again

## Summary

- **Error classes**: 400, 401, 403, 404, 408, 409, 429, 500, 503
- **Retries**: Idempotent reads with exponential backoff + jitter; writes use idempotency keys
- **Timeouts**: Auth 30s, vault 2s, NM 1s, proximity 5s
- **iOS continuations**: Always resolve (success/timeout/cancel)
- **Logging**: Structured NDJSON with `corr_id`, `origin`, `action`, `decision`, `latency_ms`, `err_code`
- **Panics**: Only on startup config errors or invariant violations; never on user input or external errors
