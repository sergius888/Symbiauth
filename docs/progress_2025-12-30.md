# Progress Log — 2025-12-30

## What’s Working Now
- **Proximity gating hardened**
  - TLS heartbeats are emitted only when iOS is actively sending traffic; if no iOS traffic for >10s, TLS sends `tls.down` and stops heartbeats.
  - Agent accepts `prox.heartbeat` **only** from TLS role; extension heartbeats are ignored.
  - Agent uses `last_heartbeat_age_ms` to determine near/far/offline; when age is missing/over grace, proximity flips to `far/offline` and vault locks.
  - `tls.down` immediately drives proximity to Far and locks vault.
- **Vault/cred gating**
  - `cred.list` / `cred.get` hard-fail when proximity is far/offline; vault locks in that state.
  - Face ID push (`auth.nudge`) is sent on session/proximity failures (when TLS is registered).
- **Web extension**
  - Popup shows proximity/vault status and heartbeat age; shows a red note when phone is away/offline.
  - Badge: blank/green when ok; red “L” when blocked/locked/far/offline.
  - Fill is blocked client-side if proximity is far/offline or vault is locked; no silent fills.
  - Softer logging for proximity_far (warn with friendly message).
  - Passive status polling every 5s keeps the popup fresh without user clicks.
- **Session replay**
  - TLS caches `vault.open` and replays it automatically after `auth.ok` (Face ID) so you don’t need to re-pair.

## Current Flow (Expected)
1) Pair + Face ID → proximity `near_unlocked`, vault `unlocked`.
2) Normal fills succeed while near and session valid.
3) If phone/iOS goes silent (airplane mode/kill app):
   - Within ~10s TLS stops heartbeats, sends `tls.down`.
   - Agent flips proximity to `far/offline`, locks vault; extension shows far/offline + locked; fills are blocked.
4) When phone/iOS returns and sends traffic, heartbeats resume, proximity returns to `near_unlocked`, vault can be unlocked again via Face ID.
